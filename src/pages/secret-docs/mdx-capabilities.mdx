# MDX Capabilities in Astro UXDS

This page summarizes the various ways you can use MDX to enhance your documentation in the Astro UXDS site.

## Overview of MDX

MDX combines the simplicity of Markdown with the flexibility of JSX. This combination allows for rich, interactive documentation that can include:

- Regular Markdown syntax for content
- Imported components from React or other frameworks
- Custom components defined directly in the document
- Web Components (custom elements) with Shadow DOM
- Interactive elements with JavaScript

## How to Use MDX

For complete documentation on using MDX in Astro UXDS, see the [Using MDX](/getting-started/using-mdx/) guide.

## When to Use MDX

MDX is particularly useful for:

1. **Interactive Documentation** - When you need to demonstrate interactive features
2. **Complex Layouts** - When you need more control over the page layout than Markdown offers
3. **Component Showcases** - When you want to demonstrate components in context
4. **Dynamic Content** - When content needs to respond to user input
5. **Consistent Formatting** - When you want to ensure consistent styling for repeated elements

## How to Choose Between MD and MDX

| Factor | Markdown (.md) | MDX (.mdx) |
|--------|--------------|-----------|
| **Simplicity** | ✅ Simpler syntax | More complex |
| **Interactivity** | Limited | ✅ Full interactivity |
| **Components** | Not supported | ✅ Fully supported |
| **Learning Curve** | ✅ Low | Higher |
| **Performance** | ✅ Slightly better | Small overhead |
| **Best For** | Simple documentation | ✅ Rich interactive content |

Choose the format that best suits your content needs. You can freely mix both formats within the same project.

## Web Component Approaches

When using web components in MDX, you have two approaches to choose from:

| Feature | React Wrapper Approach | Declarative Shadow DOM Approach |
|---------|----------------------|------------------------------|
| **Implementation** | Wrap web components in React | Use `set:html` with declarative shadow DOM |
| **Complexity** | ✅ Simpler to implement | More complex with SSR considerations |
| **SSR Compatible** | ✅ Yes, through React | ✅ Yes, through declarative shadow DOM |
| **No-JS Support** | Limited (needs hydration) | ✅ Works without JS in modern browsers |
| **Browser Support** | ✅ All browsers | ✅ Modern browsers (Chrome 90+, Safari 16.4+, Firefox 123+) |
| **Best For** | Most use cases | Performance-critical components |

> **Note:** Since 2021-2023, all major browsers have added native support for declarative shadow DOM, making it a viable option without polyfills for most developer-focused sites.

The React wrapper approach is recommended for most cases due to its simplicity and reliability. The declarative shadow DOM approach is ideal for highly optimized core components or when you want to avoid React dependencies.
